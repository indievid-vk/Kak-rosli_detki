# База Знаний: Стандарты разработки PWA (Команда Индивид СтудИИя)

Этот документ содержит проверенные решения для создания прогрессивных веб-приложений (PWA), которые стабильно работают на iOS, Android и корректно разворачиваются на GitHub Pages.

---

## 1. Принудительное окно установки (Installation UI)

Так как браузеры (особенно Safari на iOS) не всегда предлагают установку автоматически, мы используем "умное" принудительное окно.

### Логика определения устройства (`src/components/InstallPrompt.tsx`)
1. **Android/Chrome:** Слушаем событие `beforeinstallprompt`. Сохраняем событие в `state` и показываем кнопку "Установить".
2. **iOS/Safari:** Проверяем `navigator.userAgent` (наличие iPhone/iPad) и отсутствие `window.navigator.standalone`. 
3. **Общее:** Если приложение уже открыто как PWA (standalone), ничего не показываем.

### Инструкция для iOS
На iOS установка возможна только через кнопку "Поделиться" -> "На экран Домой". 
*   **Решение:** При обнаружении iOS-устройства мы показываем кастомный баннер внизу экрана с иконкой "Поделиться" и пошаговой подсказкой.

## 2. Корректный Деплой на GitHub Pages (Subfolder Routing)

Главная проблема GitHub Pages — приложение находится в подпапке (например, `...io/metanoia/`), а не в корне.

### Шаг 1: Настройка Vite (`vite.config.ts`)
Всегда используйте динамический `base`, чтобы локальная разработка (`/`) и деплой (`/metanoia/`) не конфликтовали:
```typescript
base: mode === 'production' ? '/название-репозитория/' : '/'
```

### Шаг 2: Service Worker и Относительные Пути
Чтобы SW не искал ресурсы в корне домена (что ведет к белому экрану и циклической ошибке "Обновляемся..."):
- В `manifest.json` ставьте `"start_url": "./"`.
- В `sw.js` пути к ресурсам должны начинаться с `./` или быть без слеша в начале.
- РЕГИСТРАЦИЯ: В `navigator.serviceWorker.register('sw.js')` ОПАСНО использовать `/sw.js`. Только `sw.js` (относительно `index.html`).

## 3. Стабильность иконок и PWA Manifest

Иконки часто "ломаются" при деплое на GitHub Pages, из-за чего браузер не распознает PWA и не показывает окно установки.

### Золотой стандарт: Хранение в Base64 + Скрипт восстановления
Самый надежный способ от команды «Индивид СтудИИя» (проект «Как росли детки»):
1.  **Создать текстовую копию:** Сохранить иконку как Base64 строку в файл `icon-base64.txt`. Текст в Git непотопляем.
2.  **Скрипт восстановления:** Создать `scripts/restore-icon.js`, который перед сборкой (`npm run build`) превращает текст обратно в `.png` в папке `/public`.
3.  **Автоматизация:** Добавить в `package.json` скрипт `"prebuild": "node scripts/restore-icon.js"`. 

Это на 100% гарантирует, что даже если Git «испортит» картинку при синхронизации, билд-система создаст свежую и правильную версию.

### Критический фикс: Игнорирование параметров (ignoreSearch)
Если вы используете версию в ссылках (например, `manifest.json?v=123`), Service Worker может "не узнать" файл в кэше.
*   **Решение:** Всегда используйте `{ ignoreSearch: true }` при вызове `caches.match(...)`. Это исправляет 90% проблем с "белым экраном" в оффлайне.

### Дополнительные настройки:
1.  **Относительные пути в HTML:** В `index.html` все ссылки на иконки и манифест должны быть БЕЗ начального слеша:
    *   *Ошибка:* `/icon-192.png`
    *   *Правильно:* `icon-192.png` или `./icon-192.png`
2.  **Версионирование (Cache Busting):** Чтобы GitHub Pages принудительно обновил иконки в кэшах пользователей, добавляйте версию:
    *   `<link rel="icon" href="icon-512.png?v=123">`
    *   `<link rel="manifest" href="manifest.json?v=123">`
3.  **Единообразие:** Версия (`?v=...`) в `index.html` и внутри `manifest.json` для путей к иконкам должна совпадать.

## 4. Защита от порчи файлов в Git

При работе с бинарными файлами (изображения, иконки) Git может повредить их при автоматической конвертации символов переноса строк (CRLF/LF).

### Решение: Файл `.gitattributes`
В корне проекта создайте файл `.gitattributes` и добавьте в него следующие строки. Это надежный индустриальный стандарт, гарантирующий, что Git будет обращаться с этими файлами исключительно как с бинарными данными:

```text
*.png binary
*.webp binary
*.jpg binary
*.jpeg binary
*.ico binary
```
Это предотвращает порчу файлов без необходимости исключать папку `/public/` из системы контроля версий.

## 5. Синхронизация и Экспорт данных (Zero Backend)

### Экспорт JSON (Метод Chunk-сборки)
Приложение «Как росли детки» показало, что медиафайлы могут весить сотни мегабайт. Прямая склейка строки `JSON.stringify` может уронить браузер.
*   **Лучшая практика:** Собирать массив "чанков" (кусочков строк) и создавать `Blob` напрямую из массива. Это экономит оперативную память в 2 раза.
*   **Ограничение медиа:** Установлен лимит 50Мб на один файл. Это "золотая середина" для стабильности мобильных браузеров.

## 6. Чек-лист для переноса в проект «Помощь кающимся»

1.  [ ] **Base64 Backup:** Сохранить иконки в `.txt` и добавить скрипт `restore-icon.js`.
2.  [ ] **package.json:** Прописать `"prebuild": "node scripts/restore-icon.js"`.
3.  [ ] **manifest.json:** Относительные пути (`./`) для `icons` и `start_url`.
4.  [ ] **index.html:** Ссылки на иконки и манифест без начального `/` + версия `?v=...`.
5.  [ ] **vite.config.ts:** `base: '/metanoia/'` (строго по имени репозитория).
6.  [ ] **sw.js:** Список кеша БЕЗ начальных слешей.
7.  [ ] **pwa-setup.js:** Регистрация `navigator.serviceWorker.register('sw.js')` (без слеша).
